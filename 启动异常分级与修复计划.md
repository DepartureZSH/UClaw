# UClaw 启动异常分级与修复计划

本文承接 `启动梳理文档.md`，用于统一启动加载页、Setup、日志、测试和用户支持中的异常口径。

## 分类模型

异常先按来源分类，再按严重等级分级。

### 异常类型

| 类型 | 含义 | 示例 |
|---|---|---|
| 正常阻塞 | 用户必须完成的正常流程，不算错误 | Setup 未完成、选择新工作区后等待 Setup |
| 外部异常 | 系统、磁盘、权限、路径、安全策略、网络或用户操作引起 | 数据目录不可写、App Translocation、端口被旧进程占用 |
| 内部异常 | UClaw/OpenClaw 内部流程、配置同步、Gateway、IPC 或插件状态引起 | Gateway token mismatch、Provider 缺 key、插件配置异常 |

### 异常等级

| 等级 | 名称 | 处理原则 |
|---|---|---|
| S0 | 数据风险 | 可能写错目录、损坏配置、造成数据竞争；默认阻止继续 |
| S1 | 启动阻断 | 无法安全进入主界面；停留启动页并给修复动作 |
| S2 | 功能降级 | 主界面可进入，但 AI、联网搜索、Gateway 或部分功能不可用 |
| S3 | 可恢复提示 | 正常流程提示或低风险问题，引导用户确认、重试或进入 Setup |

## 异常清单

| 异常 | 类型 | 等级 | 用户提示 | 修复动作 |
|---|---|---|---|---|
| `dataRoot` 无写入权限 | 外部异常 | S0 | 数据目录不可写，已阻止启动以避免配置丢失 | 检查磁盘权限、不要放只读目录；打开数据目录、查看日志、退出 |
| macOS App Translocation | 外部异常 | S0 | 检测到 macOS 临时转移路径，已阻止启动 | 移动 `.app` 到固定位置，必要时清理 quarantine |
| macOS `.app` 直接放 ExFAT 运行 | 外部异常 | S1 | app 运行位置不可靠，可能导致权限或签名问题 | app 放 APFS 或本机应用目录；数据目录可继续放 ExFAT |
| 移动盘盘符、卷名或挂载路径变化 | 外部异常 | S3/S2 | 工作区路径不存在，需要重新确认 | 重新选择工作区 |
| Windows SmartScreen / 杀毒拦截 | 外部异常 | S1/S2 | 系统安全策略阻止启动或写入 | 确认来源可信、加入信任、查看日志 |
| Gateway 端口被旧进程占用 | 外部异常 | S2 | Gateway 端口被占用 | 停止旧 Gateway 并重试、查看日志 |
| 网络不可用或远程 AI 服务不可达 | 外部异常 | S2 | AI 服务或联网搜索不可用 | 重新扫描配置、切换 Provider、检查网络或代理 |
| 多系统同时打开同一移动盘数据 | 外部异常 | S0 | 数据目录可能被并发写入 | 阻止第二个实例写入，关闭另一端后重试 |
| 拔盘或同步未完成导致 JSON 损坏 | 外部异常 | S0/S1 | 启动配置文件可能已损坏 | 先备份 dataRoot/workspace，查看日志定位损坏文件 |
| Setup 未完成 | 正常阻塞 | S3 | 需要先完成 Setup | 进入 Setup |
| `setupComplete=true` 但工作区为空 | 正常阻塞 | S3 | 工作区需要重新确认 | 自动重置 Setup 状态 |
| Provider store 与 runtime auth 不一致 | 内部异常 | S2 | Provider 配置需要同步 | 重新同步 token、重新扫描配置 |
| 默认 Provider 缺 key | 内部异常 | S2 | 默认 Provider 缺少可用密钥 | 切换 Provider、重新扫描配置 |
| Gateway token mismatch / unauthorized | 内部异常 | S2/S1 | Gateway 认证失败 | 重新同步 token、重启 Gateway |
| Gateway ready / RPC timeout | 内部异常 | S1 | Gateway 响应超时 | 重启 Gateway、查看日志 |
| 插件配置异常或缺插件 | 内部异常 | S2/S1 | Gateway 插件配置异常 | 查看日志、清理异常插件配置、重试 |
| 启动状态机未从 Setup 回到 ready | 内部异常 | S1 | Setup 后仍停留启动页或 Setup | Setup 完成后等待 `retry-current-step` 返回并更新 startup store |
| 意外迁移用户本机 `~/.openclaw` | 内部异常 | S0 | 可能污染用户原生 OpenClaw 配置 | 默认禁止迁移，单测锁定不导入 standalone OpenClaw config |

## 当前已实现

- `StartupIssueType = external | internal | normal-blocking`
- `StartupSeverity = S0 | S1 | S2 | S3`
- `StartupSnapshot.issue` 携带异常类型、等级、代码、标题和修复建议。
- 启动页按 `S0-S3` 显示不同风险提示。
- 新增统一错误修复页，覆盖 React 渲染错误、全局未处理异常、macOS App Translocation 和极早期 dataRoot 初始化失败。
- 已知界面错误会分类为 `RENDER_CHUNK_LOAD_FAILED`、`IPC_CHANNEL_UNAVAILABLE`、`RENDER_CACHE_QUOTA_EXCEEDED`、`NETWORK_OR_LOCAL_SERVICE_UNAVAILABLE`、`SYSTEM_PERMISSION_DENIED`、`RENDER_LAYOUT_RECOVERABLE`、`RENDER_UNKNOWN_ERROR`。
- 已支持动作：`retry-current-step`、`restart-gateway`、`stop-old-gateway-and-retry`、`resync-token`、`switch-provider`、`rescan-provider-config`、`select-workspace`、`open-workspace-folder`、`open-data-root`、`open-log-folder`、`copy-diagnostics`、`quit-app`。
- 错误修复页额外支持：重试页面、重新加载页面、重启 UClaw、重启 Gateway、打开日志目录、打开数据目录、复制诊断信息、清理页面临时缓存并重载。
- `dataRoot` 启动时执行写入预检。
- `dataRoot` 写入前会创建 `<dataRoot>/uclaw/data-root.lock`，阻止多系统或多进程同时写同一个数据目录。
- `openclaw.json` 解析失败时会保留 `.corrupt.*` 副本并阻止继续写回，避免把损坏文件覆盖成空配置。
- Gateway timeout 已细分为进程启动失败、端口等待超时、WebSocket 握手超时、RPC ready 超时。
- 复制诊断信息已包含 dataRoot、uclawDir、openclawDir、workspaceDir、settingsPath、providerStorePath、logDir、当前步骤和 issue code。
- 默认不迁移用户本机 `~/.openclaw`。

## 后续修复计划

### P0 数据安全

1. 增强数据目录锁的阻断页体验，提示用户如何判断锁是否为陈旧锁。
2. 扩展 JSON parse 保护到更多启动期 JSON 文件，例如 provider store、auth profiles 和插件 manifest。
3. 增加导出完整诊断包能力，便于支持人员一次性收集日志、配置路径和异常快照。

### P1 启动阻断

1. 完善 macOS ExFAT app 运行位置诊断，文案明确“数据可放 ExFAT，app 不建议放 ExFAT”。
2. 增加 Gateway 子阶段错误的日志定位提示，把失败阶段和最近一条 Gateway stderr 一并带入诊断。
3. 增加更完整的诊断复制内容：最近启动动作、修复动作结果、Gateway 状态摘要。

### P2 功能降级

1. Provider/key warning 保持在启动页内修复，不跳 Settings。
2. 网络或远程 AI 服务不可达时，显示“功能降级但可进入主界面”的提示。
3. 插件异常增加更具体的日志定位建议。

### P3 普通提示

1. Setup 未完成、工作区不存在、路径变化保持可恢复提示。
2. 用户手册继续使用普通用户语言，不暴露内部 IPC 或环境变量细节。
3. E2E 覆盖首次 Setup、再次打开、跨工作区路径变化和 Provider 缺 key。
