# UClaw 启动流程梳理

本文面向产品、测试、支持和交付人员，梳理普通用户从安装、首次打开、再次打开，到跨操作系统使用 UClaw 时的实际流程、可能异常和涉及的配置文件。

异常类型、严重等级、启动页文案和修复动作统一见：[启动异常分级与修复计划.md](./启动异常分级与修复计划.md)。

## 统一规则

UClaw 现在不再区分“普通安装版 / 便携版”双模式。应用只关心一个明确的数据根目录：`dataRoot`。

- 普通安装版：默认使用系统应用数据目录。
- 移动盘 / USB 版：由平台启动器显式传入 `--uclaw-data-root=<移动盘>/data`。
- E2E 测试：同样通过 `--uclaw-data-root=<临时目录>/data` 指定数据目录。
- UClaw 不会自动迁移用户本机自己的 `~/.openclaw`。

默认数据结构：

```text
<dataRoot>/
├── uclaw/
│   ├── settings.json
│   ├── uclaw-providers.json
│   ├── logs/
│   └── .storage-migrated
└── .openclaw/
    ├── openclaw.json
    └── agents/
        └── main/
            └── agent/
                ├── auth-profiles.json
                └── models.json
```

如果用户在 Setup 中选择了工作区，OpenClaw 运行时配置优先放在：

```text
<workspaceDir>/.openclaw/openclaw.json
<workspaceDir>/.openclaw/agents/main/agent/auth-profiles.json
<workspaceDir>/.openclaw/agents/main/agent/models.json
```

但 UClaw 自身设置仍然在：

```text
<dataRoot>/uclaw/settings.json
<dataRoot>/uclaw/uclaw-providers.json
```

## 启动加载页阶段

每次打开 UClaw 都会先经过启动加载页，由 Main 进程统一编排以下阶段：

1. `app-init`：应用初始化，确定 `dataRoot`。
2. `settings-load`：读取 UClaw 设置。
3. `workspace-resolve`：解析工作区。
4. `setup-check`：判断是否完成 Setup。
5. `config-sync`：同步启动配置。
6. `provider-key-sync`：检查 Provider 和 API Key。
7. `gateway-start`：启动 OpenClaw Gateway。

状态含义：

- `ready`：启动完成，进入主界面。
- `blockedBySetup`：Setup 未完成，进入 Setup。
- `warning`：可以继续启动，但存在 Provider/key 等需要用户处理的问题。
- `error` / `timeout`：停留在启动加载页，显示中文错误、技术详情和操作按钮。

## 1. 安装后第一次打开

### 用户操作

1. 安装或解压 UClaw。
2. 打开应用。
3. 看到启动加载页。
4. 如果 Setup 未完成，进入 Setup。
5. 选择工作区。
6. 使用 UClaw 提供的一键 AI 服务配置。
7. 完成后返回启动流程，Gateway ready 后进入主界面。

### 涉及配置文件

UClaw 自身配置：

```text
<dataRoot>/uclaw/settings.json
<dataRoot>/uclaw/uclaw-providers.json
<dataRoot>/uclaw/logs/
<dataRoot>/uclaw/.storage-migrated
```

默认 OpenClaw 运行时配置：

```text
<dataRoot>/.openclaw/openclaw.json
<dataRoot>/.openclaw/agents/main/agent/auth-profiles.json
<dataRoot>/.openclaw/agents/main/agent/models.json
```

如果选择了工作区：

```text
<workspaceDir>/.openclaw/openclaw.json
<workspaceDir>/.openclaw/agents/main/agent/auth-profiles.json
<workspaceDir>/.openclaw/agents/main/agent/models.json
```

### 可能异常

- `dataRoot` 无写入权限：无法保存设置或 Provider。
- Setup 未完成：进入 Setup，这是正常阻塞状态，不算错误。
- Provider/API Key 缺失：启动页显示 warning，可在启动页内修复。
- Gateway 端口占用：启动页提示停止旧 Gateway 后重试。
- Gateway 启动超时：启动页停留并提供重试、日志和退出入口。
- macOS App Translocation：启动被阻止，提示修复启动方式。

## 2. 再次打开

### 用户操作

1. 打开 UClaw。
2. 启动加载页读取设置、工作区、Provider、Gateway 状态。
3. Gateway ready 后进入主界面。

### 主要读取文件

UClaw 设置：

```text
<dataRoot>/uclaw/settings.json
<dataRoot>/uclaw/uclaw-providers.json
```

OpenClaw 配置二选一：

```text
<dataRoot>/.openclaw/openclaw.json
```

或：

```text
<workspaceDir>/.openclaw/openclaw.json
```

运行时还会读取：

```text
.../.openclaw/agents/main/agent/auth-profiles.json
.../.openclaw/agents/main/agent/models.json
.../.openclaw/agents/<agentId>/sessions/*.jsonl
```

### 可能异常

- `settings.json` 里的 `workspaceDir` 不存在：重置 Setup 状态，要求重新选择工作区。
- `setupComplete=true` 但工作区为空：重置为 Setup。
- `uclaw-providers.json` 有账号但密钥缺失：Provider 检查 warning。
- `auth-profiles.json` 和 Provider store 不一致：启动时尝试同步修复。
- Gateway token 不匹配：启动页提示重新同步 token。
- 旧 Gateway 仍在运行：端口占用，提示停止旧进程。

## 3. Windows 换到 macOS 第一次打开

### 普通安装版换系统

用户操作：

1. 在 macOS 安装 UClaw。
2. macOS 默认使用自己的系统 `dataRoot`。
3. 不会自动读取 Windows 机器上的配置。
4. 第一次打开会进入 Setup。

macOS 新文件通常位于：

```text
~/Library/Application Support/uclaw/settings.json
~/Library/Application Support/uclaw/uclaw-providers.json
~/Library/Application Support/.openclaw/openclaw.json
```

### 移动盘跨系统

用户操作：

1. Windows 上已使用移动盘版。
2. macOS 上不要直接从 ExFAT 运行 `.app`。
3. 推荐 `.app` 放 APFS 分区，数据和工作区放 ExFAT 分区。
4. 用 `Launch UClaw.command` 启动。
5. 启动器传入共享数据目录。

示例：

```bash
open "$APP" --args --uclaw-data-root /Volumes/SHARE_EXFAT/data
```

共享文件继续使用：

```text
/Volumes/SHARE_EXFAT/data/uclaw/settings.json
/Volumes/SHARE_EXFAT/data/uclaw/uclaw-providers.json
/Volumes/SHARE_EXFAT/data/.openclaw/openclaw.json
/Volumes/SHARE_EXFAT/workspace/.openclaw/openclaw.json
```

### 可能异常

- macOS App Translocation：用 Finder 直接打开或隔离属性未清理，启动页会阻止继续。
- ExFAT 不适合直接运行 `.app`：可能出现权限、签名、软链异常。
- 卷名变化：`/Volumes/SHARE_EXFAT` 路径变了，启动器需要对应更新。
- 工作区路径不存在：进入 Setup 重新选择。
- Keychain 不共享：如果某些密钥曾走系统钥匙串，跨 OS 可能需要重新同步。

## 4. macOS 换到 Windows 第一次打开

### 普通安装版换系统

用户操作：

1. 在 Windows 安装 UClaw。
2. Windows 使用自己的默认 `dataRoot`。
3. 不读取 macOS 的系统配置。
4. 第一次打开进入 Setup。

Windows 默认相关文件类似：

```text
%APPDATA%\uclaw\settings.json
%APPDATA%\uclaw\uclaw-providers.json
%APPDATA%\.openclaw\openclaw.json
```

### 移动盘跨系统

用户操作：

1. 使用 Windows 启动器，例如 `Launch UClaw Windows.cmd`。
2. 启动器传入共享数据目录。
3. 如果之前 macOS 使用的是同一个共享 `dataRoot` 和 workspace，Windows 会读取同一套配置。
4. 启动加载页检查后直接进入主界面，或要求重新确认工作区。

示例：

```cmd
windows\UClaw.exe --uclaw-data-root "%SCRIPT_DIR%data"
```

涉及文件：

```text
<USB>\data\uclaw\settings.json
<USB>\data\uclaw\uclaw-providers.json
<USB>\data\.openclaw\openclaw.json
<USB>\workspace\.openclaw\openclaw.json
```

### 可能异常

- Windows SmartScreen：需要用户确认来源可信。
- 盘符变化：例如从 `E:` 变成 `F:`，如果配置里记录了绝对 workspace 路径，可能触发工作区不存在。
- 路径大小写或分隔符差异通常可处理，但旧绝对路径仍可能失效。
- Gateway 端口被 Windows 上已有进程占用：启动页提示处理。
- 杀毒软件拦截可执行文件或写入 `data`：表现为启动失败或配置无法保存。

## 5. 多次更换操作系统打开

### 推荐使用方式

1. 始终用各平台包内启动器启动。
2. 始终让启动器指向同一个共享 `dataRoot`。
3. 始终把工作区放在移动盘的稳定位置。
4. 不手动编辑 `openclaw.json` 里的 key。
5. 不把 macOS `.app` 放在 ExFAT 上直接运行。

稳定目录建议：

```text
SHARE_EXFAT/
├── data/
│   ├── uclaw/
│   │   ├── settings.json
│   │   └── uclaw-providers.json
│   └── .openclaw/
│       └── openclaw.json
└── workspace/
    └── .openclaw/
        └── openclaw.json
```

### 常见异常

- 移动盘挂载路径变化：Windows 盘符变、macOS 卷名变、Linux mount path 变。
- 工作区绝对路径失效：启动页进入 Setup，要求重新选择工作区。
- macOS 隔离 / App Translocation：需要用启动器或清理 quarantine。
- 文件同步未完成就拔盘：JSON 可能损坏，启动页或 Setup 会暴露异常。
- 多系统同时打开同一个移动盘数据：可能导致配置竞争，不建议。
- Provider key 与 runtime auth profile 不一致：启动时会尝试同步，失败则给 warning。
- Gateway 端口、旧进程、权限问题：停在启动页，提供重试、日志、退出操作。

## 总结

普通安装版是“每个系统各自一套 `dataRoot`”。

移动盘跨系统是“每个平台启动器都指向同一个 `dataRoot`”。

UClaw 自己只迁移和管理 `<dataRoot>/uclaw`，不会擅自搬用户本机 `~/.openclaw`。
