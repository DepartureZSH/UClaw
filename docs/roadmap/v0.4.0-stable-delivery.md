# UClaw v0.4.0 稳定性交付规划

## 目标

v0.4.0 的目标是把 UClaw 商业 ZIP 版打磨成可交付的 ToB 随盘 AI 工作台：制作人员配置一次，普通用户在不同电脑上即插即用；启动、更新、OpenClaw runtime、频道连接和错误修复都形成闭环。

一句话验收标准：

> 客户拿到已初始化的 UClaw 移动盘后，不需要理解工作区、API Key、OpenClaw、Gateway 或配置文件；如果遇到问题，应用能给出明确原因、一键修复动作和可脱敏发送给售后的诊断包。

## 产品边界

### v0.4.0 做

- 固定随盘工作台稳定化。
- 制作人员初始化模式与普通用户使用模式分离。
- 普通用户免 Setup、免 API 配置、免工作区选择。
- 启动和运行错误统一进入错误修复页。
- 一键诊断包。
- ZIP 便携更新流程设计并落地第一版。
- OpenClaw runtime 检查、更新和回滚第一版。
- 频道健康检查和一键修复第一版。

### v0.4.0 不做

- 完整企业后台。
- 设备级授权和复杂组织权限。
- 企业知识库、RAG 和行业工作流。
- 用量计费与成本报表。
- 私有化部署控制台。

这些能力放入 v0.5.0 之后，避免 v0.4.0 失焦。

## 用户角色

| 角色 | 目标 | 应看到的能力 | 不应看到的内容 |
|---|---|---|---|
| 普通用户 | 打开即可对话和使用频道 | 启动页、对话、频道状态、更新提示、错误修复、复制诊断 | API URL、API Key、工作区选择、OpenClaw 配置、Gateway 技术细节 |
| 制作人员 | 初始化移动盘并完成交付检查 | 公司密钥、packageId、远程配置同步、工作区初始化、交付检查报告 | Laf PAT、生产 API Key 明文、源码级调试入口 |
| 售后人员 | 快速定位客户现场问题 | 诊断包、错误等级、修复动作结果、版本和路径摘要 | 未脱敏密钥、用户隐私内容全文 |
| 开发人员 | 维护和升级产品 | 开发者模式、日志、Doctor、测试入口 | 普通用户主流程中的开发者配置 |

## 核心原则

1. 普通用户流程不出现 Setup。
2. 普通用户不填写 API 地址或 API Key。
3. 随盘路径必须相对化，不能被 Windows 盘符、macOS 挂载点或本机 Roaming 污染。
4. 新配置写入必须可回滚，不能覆盖最后一次可用配置。
5. 更新系统必须区分应用更新、OpenClaw runtime 更新和远程配置更新。
6. 所有一键修复动作都要写日志，并能进入诊断包。
7. 任何诊断输出默认脱敏。

## 架构分层

### 1. Workbench Layer

负责固定随盘数据根、工作区、OpenClaw state 和 runtime 路径。

关键行为：

- ZIP 版默认使用程序旁 `data/`。
- 工作区固定解析为 `<dataRoot>/workspace`。
- OpenClaw state 固定解析为 `<dataRoot>/workspace/.openclaw`。
- 不读取、不迁移、不复用宿主机 Roaming 或用户本机 `~/.openclaw`。
- 发现相对工作区缺失时优先自动重建基础目录，而不是回到 Setup。

### 2. Provisioning Layer

负责公司密钥、packageId、Laf 配置下发和本地缓存。

关键行为：

- 制作人员输入公司密钥后，UClaw 换取 package 级运行配置。
- 运行配置写入本地 provider store 和 OpenClaw auth，不把 API Key 明文写入 `openclaw.json`。
- Laf 不可达时使用最后一次有效缓存。
- 无有效缓存且无法同步时进入 S1 阻断修复页。
- 配置被吊销时进入 S1 阻断修复页，并显示停用原因。

### 3. Startup and Repair Layer

负责启动阶段、错误分类和修复页。

关键行为：

- 启动阶段必须覆盖：app-init、settings-load、remote-config-sync、workspace-repair、provider-key-sync、runtime-check、gateway-start。
- S0/S1 阻止进入主界面，必须给出修复动作。
- S2 允许进入主界面，但显示降级提示。
- S3 只做确认或提示，不当作错误。
- 修复动作包括重试配置同步、重建工作区、重新生成 OpenClaw 配置、重启 Gateway、打开日志、复制诊断、退出应用。

### 4. Update Layer

负责 ZIP 便携版应用更新和 OpenClaw runtime 更新。

关键行为：

- ZIP 便携版不使用 `electron-updater` 的安装器自动更新。
- 应用 ZIP 更新流程为：检查版本、下载 ZIP、校验 hash、解压到 staging、退出后替换、失败回滚。
- OpenClaw runtime 单独维护版本、下载、校验、替换和回滚。
- 远程配置更新只更新 provider、web-search、模型策略和公告，不替换程序文件。

### 5. Channel Reliability Layer

负责微信、飞书、钉钉、Telegram、WhatsApp 等频道的健康检查和修复。

关键行为：

- 每个频道展示登录状态、插件状态、Agent 绑定状态、Gateway 可达状态和最近错误。
- 每个频道支持发送测试消息。
- 二维码类频道支持重新扫码和重建会话目录。
- Webhook/token 类频道支持重新验证凭证。
- 频道修复动作完成后必须重启或热刷新 Gateway，并记录结果。

### 6. Diagnostics Layer

负责售后诊断包。

诊断包必须包含：

- UClaw 版本、平台、架构、打包类型。
- dataRoot、workspaceDir、openclawDir、logDir。
- 公司配置版本、packageId 摘要、配置缓存状态。
- Provider 和 web-search 配置状态。
- Gateway 状态、端口、pid、最近错误阶段。
- OpenClaw runtime 版本和路径。
- 频道健康摘要。
- 最近日志片段。
- 用户触发过的修复动作和结果。

诊断包必须脱敏：

- API Key。
- 公司密钥。
- token。
- cookie。
- OAuth credential。
- 个人消息正文默认不进入诊断包。

## v0.4.0 里程碑

### Milestone 1: 随盘工作台和普通用户免配置

目标：已初始化移动盘在不同 Windows/macOS/Linux 路径下启动，不回到 Setup。

交付项：

- 相对工作区解析稳定。
- 缺失基础目录时自动修复。
- 普通用户流程隐藏 Setup。
- 公司密钥页只在缺少 package credential 或配置被吊销时出现。
- 启动页文案不暴露开发者术语。

验收：

- 同一份 `data/` 在不同盘符启动，不重新配置。
- 删除空的 `data/workspace/.openclaw` 后可自动重建。
- 存在有效远程配置缓存时，断网仍可进入主界面。
- 没有有效配置且 Laf 不可达时，停留 S1 修复页。

### Milestone 2: 一键诊断包和统一错误修复页

目标：售后不需要让用户翻日志，用户能一键复制或导出诊断信息。

交付项：

- 诊断包生成接口。
- 诊断脱敏器。
- 错误修复页接入诊断包。
- 修复动作结果记录。
- 启动页和运行期错误页共用异常分类。

验收：

- 诊断包不包含 `sk-` 格式密钥、公司密钥、token 或 cookie。
- Gateway timeout、远程配置失败、dataRoot 不可写、频道连接失败都能生成诊断包。
- 修复动作失败时，诊断包包含失败动作和错误摘要。

### Milestone 3: ZIP 便携更新器

目标：ZIP 版本能安全提示、下载和准备更新，不再误用安装器更新。

交付项：

- ZIP 更新 manifest 解析。
- ZIP 下载和 hash 校验。
- `data/uclaw/update-staging` 暂存目录。
- 退出后替换或下次启动替换。
- 失败回滚和清理。
- 更新页区分“应用 ZIP 更新”和“OpenClaw runtime 更新”。

验收：

- 没有 `app-update.yml` 时不报 ENOENT。
- ZIP 更新失败不破坏当前可运行版本。
- staging 中断后下次启动能继续清理或回滚。
- 慢速 U 盘上更新页明确提示建议先在本机 SSD 解压再整体复制。

### Milestone 4: OpenClaw runtime 更新和回滚

目标：UClaw 能独立检查和修复 OpenClaw runtime，而不是只能随应用升级。

交付项：

- runtime 版本检测。
- runtime 包下载和校验。
- runtime 切换为 versioned layout。
- 上一版本保留。
- Gateway 启动失败时可回滚 runtime。
- Doctor fix 可优先修 runtime。

验收：

- runtime 损坏时能进入 S1 修复页并提供重装 runtime。
- 新 runtime 启动失败时能回滚到上一版本。
- 回滚不覆盖用户工作区和 provider/auth。

### Milestone 5: 频道健康检查和一键修复

目标：频道从“能配置”升级为“可运维”。

交付项：

- 频道健康检查接口。
- 渠道页健康状态展示。
- 测试消息动作。
- 二维码频道重新扫码。
- webhook/token 频道重新验证。
- Agent 绑定检查。

验收：

- 微信扫码成功但无法连接 OpenClaw 时，页面能指出失败在 Gateway、Agent 绑定、插件或会话状态哪一层。
- 每个已配置频道都能生成健康摘要进入诊断包。
- 一键修复动作后有成功/失败反馈。

## 推荐实施顺序

1. 先做 Milestone 1，解决普通用户不应重新 Setup 的商业底线。
2. 再做 Milestone 2，让所有后续问题都有诊断出口。
3. 接着做 Milestone 3 和 4，完成应用与 runtime 更新闭环。
4. 最后做 Milestone 5，把频道问题纳入同一套可诊断体系。

## 测试策略

### Unit

- 相对工作区解析。
- dataRoot/workspace 自动修复。
- 远程配置缓存成功、失败、吊销。
- 诊断脱敏。
- ZIP 更新状态机。
- runtime 回滚状态机。
- 频道健康分类。

### Main

- 启动阶段快照。
- Provider/auth 写入。
- OpenClaw runtime env 注入。
- Gateway 子阶段错误分类。
- 修复动作日志记录。

### Renderer

- 公司密钥页。
- 启动页。
- 错误修复页。
- 诊断包复制和导出。
- 更新页。
- 频道健康状态。

### E2E

- 制作人员初始化一份移动盘数据。
- 同一份数据在不同盘符或挂载路径启动。
- Laf 不可达但有缓存。
- Laf 不可达且无缓存。
- OpenClaw runtime 损坏后重装。
- ZIP 更新下载失败后回滚。
- 微信/飞书/钉钉至少一个频道进入健康检查。

## 发布门槛

v0.4.0 发布前必须满足：

- `pnpm run typecheck` 通过。
- 相关 Vitest 通过。
- 受影响 Electron E2E 通过。
- 通信路径变更时 `pnpm run comms:replay` 和 `pnpm run comms:compare` 通过。
- Windows ZIP、macOS ZIP、Linux ZIP 都能解压启动。
- Windows ZIP 在新电脑上不进入 Setup。
- 缺少公司密钥时只进入公司密钥页，不进入旧 Setup。
- 诊断包脱敏测试通过。
