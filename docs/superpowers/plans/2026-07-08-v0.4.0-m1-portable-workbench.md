# v0.4.0 Milestone 1 Portable Workbench Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make initialized commercial ZIP workbenches open on different machines without showing legacy Setup, workspace selection, API URL, or API Key configuration to ordinary users.

**Architecture:** Keep data-root and portable marker resolution in Main, make the startup state machine explicitly repair and publish portable workspace state, and keep Renderer route gating dependent only on startup snapshots. The implementation is incremental: first lock path behavior with unit tests, then expose clearer startup steps/actions, then cover ordinary-user flows with renderer and E2E tests.

**Tech Stack:** Electron Main, React 19, TypeScript, Vitest, Playwright Electron E2E, existing `host-api`/IPC wrappers.

---

## File Map

- Modify `shared/startup.ts`: add `workspace-repair` step label and preserve existing startup snapshot contract.
- Modify `electron/main/workspace-startup.ts`: return richer portable workbench repair metadata and never reset Setup in portable mode.
- Modify `electron/main/startup-progress-service.ts`: split workspace repair from legacy Setup compatibility and improve ordinary-user messages/actions.
- Modify `src/App.tsx`: keep `/setup` as a company-key compatibility route before ready, and prevent legacy Setup from reappearing after ready in portable workbench mode.
- Modify `src/pages/CompanyKeyPage.tsx`: make the page copy production-oriented and explicitly for company-key provisioning only.
- Modify `tests/unit/workspace-startup.test.ts`: lock cross-drive and missing-directory behavior.
- Modify `tests/unit/startup-progress-service.test.ts`: lock step order, company-key blocking, and portable repair warning/error behavior.
- Create `tests/e2e/portable-workbench.spec.ts`: simulate a prepared ZIP data directory moving across machine paths.
- Review docs after implementation: `README.md`, `README.zh-CN.md`, `README.ja-JP.md`, `user-guide.md`, `docs/zh-CN/advanced/portable-usb.md`.

## Acceptance Criteria

- A v2 `uclaw-portable.json` with `workspaceMode: "portable-workbench"` always resolves workspace as `<dataRoot>/<workspaceDir>`.
- Stored `settings.json.workspaceDir` remains relative, for example `workspace`, after startup.
- A stale absolute workspace path from another computer is overwritten in portable mode and does not trigger Setup.
- Missing `<dataRoot>/workspace/.openclaw` is automatically recreated when the data root is writable.
- Missing company configuration shows only the company-key page, not the legacy Setup wizard.
- After startup reaches `ready`, normal routes render the main app; `/setup/*` redirects or renders the company-key compatibility page, not the old setup wizard.
- If portable workspace repair fails, startup stops with `PORTABLE_WORKSPACE_REPAIR_FAILED` and actions to open data/log folders and copy diagnostics.

Create `tests/e2e/portable-workbench.spec.ts`:

```ts
import { test, expect } from '@playwright/test';
import { mkdtemp, mkdir, writeFile, cp, rm } from 'node:fs/promises';
import { join } from 'node:path';
import { tmpdir } from 'node:os';

test.describe('portable workbench startup', () => {
  test('keeps initialized portable workspace across different parent paths', async () => {
    const rootA = await mkdtemp(join(tmpdir(), 'uclaw-portable-a-'));
    const rootB = await mkdtemp(join(tmpdir(), 'uclaw-portable-b-'));
    try {
      const dataA = join(rootA, 'data');
      await mkdir(join(dataA, 'uclaw'), { recursive: true });
      await mkdir(join(dataA, 'workspace', '.openclaw'), { recursive: true });
      await writeFile(join(dataA, 'uclaw', 'settings.json'), JSON.stringify({
        setupComplete: true,
        workspaceDir: 'workspace',
        gatewayAutoStart: false,
      }, null, 2), 'utf8');
      await writeFile(join(dataA, 'workspace', '.openclaw', 'openclaw.json'), JSON.stringify({
        models: { default: 'new-api/deepseek-v4-flash' },
      }, null, 2), 'utf8');

      const dataB = join(rootB, 'data');
      await cp(dataA, dataB, { recursive: true });

      expect(dataB).not.toBe(dataA);
      const copiedSettings = await import('node:fs/promises').then((fs) =>
        fs.readFile(join(dataB, 'uclaw', 'settings.json'), 'utf8'),
      );
      expect(copiedSettings).toContain('"workspaceDir": "workspace"');
    } finally {
      await rm(rootA, { recursive: true, force: true });
      await rm(rootB, { recursive: true, force: true });
    }
  });
});
```

This initial E2E is file-level and intentionally cheap. If the existing Electron E2E harness exposes an app launcher with custom env/args, replace the file-only assertion with launching the app using `--uclaw-data-root <rootB>/data` and asserting the startup page reaches ready without legacy Setup.

- After this milestone lands, Milestone 2 can build diagnostics on top of the richer workspace/startup metadata.
